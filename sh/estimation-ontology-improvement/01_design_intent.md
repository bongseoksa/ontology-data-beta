# 01. Schema Design Intent

## 1) 문제 정의

현재 구조는 계산식 중심으로 잘 정리되어 있으나, 아래 운영 문제가 발생할 수 있습니다.

- 산정 근거 추적 불가: 값은 있는데 왜 그 값인지 재현이 어려움
- 입력 품질 불안정: 요구사항 누락/모호성 상태에서 견적 생성 가능
- 점추정 편향: 단일 값만 저장되어 리스크 대응이 약함
- 버전 혼선: 단가표/정책표/가이드 개정 시 과거 견적 재해석이 어려움

## 2) 설계 목표

1. 설명가능성
- 견적 수치가 어떤 요구사항/근거문서/단가표에서 왔는지 추적 가능

2. 검증가능성
- 견적 생성 전에 필수 입력과 일관성을 SHACL로 자동 검증

3. 확장가능성
- FP, COSMIC, COCOMO, 내부 생산성 모델을 공존시키는 모듈 구조

4. 운영성
- 정책/단가/모형 버전을 분리하여 변경 영향 추적

## 3) 핵심 설계 결정

1. 모듈 분리
- `req`, `size`, `cost`, `risk`, `price`, `quote`, `prov`로 분할

2. 시나리오-버전 모델
- `quote:Quote`는 비즈니스 식별자
- `quote:QuoteVersion`은 실제 계산 스냅샷

3. 범위 견적 모델
- `quote:EstimateRange(min/p50/p80/max)`를 기본 저장 단위로 사용

4. 증거 연결
- `prov:wasDerivedFrom`로 RFP, 가이드, 단가표, 데이터셋을 연결

## 4) 비목표(Non-goals)

- 특정 벤더 트리플스토어 종속 설계
- 단일 추정기법 강제 (FP만 강제하지 않음)
- 완전 자동 견적(사람 검토 단계는 유지)

## 5) 공유 시 합의 포인트

- 어떤 필드는 "없으면 견적 불가"인지 (Hard Validation)
- 어떤 필드는 "없어도 견적 가능하나 신뢰도 하락"인지 (Soft Validation)
- 공공 프로젝트와 민간 프로젝트 정책 분기 기준
