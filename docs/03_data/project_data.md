# Phase 3-2: 프로젝트 구조 데이터 소스

## 1. 기능점수(Function Point) 산정 가이드

### 1.1 공식 표준 및 가이드

| 표준/가이드 | 발행처 | 설명 |
|------------|--------|------|
| **ISO/IEC 20926** | ISO | 국제 FP 산정 표준 |
| **IFPUG CPM** | IFPUG | FP 측정 실무 매뉴얼 |
| **간이법 FP 가이드** | KOSA | 한국형 간이 FP 산정 |
| **SW사업 FP 가이드** | 과기정통부 | 공공사업 FP 적용 |

### 1.2 FP 구성 요소

```
기능점수(FP) = 데이터기능 + 트랜잭션기능

[데이터 기능]
├── ILF (내부논리파일): 시스템 내 관리 데이터
└── EIF (외부연계파일): 외부 참조 데이터

[트랜잭션 기능]
├── EI (외부입력): 데이터 입력/수정/삭제
├── EO (외부출력): 파생 데이터 출력
└── EQ (외부조회): 단순 데이터 조회
```

### 1.3 기능 유형별 복잡도 가중치

| 기능 유형 | 단순 | 보통 | 복잡 |
|----------|:----:|:----:|:----:|
| ILF | 7 | 10 | 15 |
| EIF | 5 | 7 | 10 |
| EI | 3 | 4 | 6 |
| EO | 4 | 5 | 7 |
| EQ | 3 | 4 | 6 |

### 1.4 FP 데이터 활용 방법

| 활용처 | 방법 |
|--------|------|
| **개발 규모 추정** | 요구사항 → FP 매핑 |
| **비용 산정** | FP × FP당 단가 |
| **생산성 비교** | FP / MM (투입공수) |

## 2. SW개발 방법론 표준

### 2.1 주요 방법론별 특성

| 방법론 | 특성 | 비용 영향 요소 |
|--------|------|---------------|
| **Waterfall** | 순차적, 문서 중심 | 초기 설계 비용 높음 |
| **Agile/Scrum** | 반복적, 유연함 | 변경 대응 비용 낮음 |
| **DevOps** | 자동화, 지속 배포 | 초기 인프라 비용 높음 |
| **SAFe** | 대규모 Agile | 조직 구성 비용 높음 |

### 2.2 방법론별 투입 인력 구조

```
[Waterfall 프로젝트]
├── PM: 전체 기간
├── 분석가: 초기 30%
├── 설계자: 초기~중기 40%
├── 개발자: 중기 50%
└── 테스터: 후기 30%

[Agile/Scrum 프로젝트]
├── Scrum Master: 전체 기간
├── Product Owner: 전체 기간 (부분 투입)
├── 개발팀: 전체 기간 (Cross-functional)
└── QA: 전체 기간 (통합)
```

### 2.3 방법론 관련 데이터 소스

| 데이터 | 소스 | 활용 |
|--------|------|------|
| Scrum 가이드 | scrum.org | 역할 정의 |
| SAFe 프레임워크 | scaledagileframework.com | 대규모 조직 |
| PMBOK | PMI | 전통적 PM |

## 3. 프로젝트 WBS 템플릿

### 3.1 소프트웨어 프로젝트 표준 WBS

```
1. 프로젝트 관리
   1.1 착수
   1.2 계획 수립
   1.3 진행 관리
   1.4 종료

2. 요구사항 분석
   2.1 요구사항 수집
   2.2 요구사항 분석
   2.3 요구사항 명세
   2.4 요구사항 검증

3. 설계
   3.1 아키텍처 설계
   3.2 상세 설계
   3.3 UI/UX 설계
   3.4 데이터베이스 설계

4. 개발
   4.1 환경 구축
   4.2 프론트엔드 개발
   4.3 백엔드 개발
   4.4 통합

5. 테스트
   5.1 단위 테스트
   5.2 통합 테스트
   5.3 시스템 테스트
   5.4 인수 테스트

6. 배포 및 이관
   6.1 배포 계획
   6.2 데이터 마이그레이션
   6.3 사용자 교육
   6.4 운영 이관
```

### 3.2 WBS 단계별 투입 비율 (표준)

| 단계 | 비율 | 주요 역할 |
|------|:----:|----------|
| 프로젝트 관리 | 10~15% | PM |
| 요구사항 분석 | 10~15% | BA, 아키텍트 |
| 설계 | 15~20% | 아키텍트, 설계자 |
| 개발 | 35~45% | 개발자 |
| 테스트 | 15~20% | QA, 테스터 |
| 배포/이관 | 5~10% | DevOps, PM |

## 5. 복잡도 평가 기준

### 5.1 기술 복잡도 요소

| 요소 | 저 | 중 | 고 |
|------|:--:|:--:|:--:|
| 동시 사용자 수 | <100 | 100~1000 | >1000 |
| 데이터 규모 | GB | 수십GB | TB+ |
| 외부 연동 수 | 1~2 | 3~5 | 6+ |
| 실시간 처리 | 없음 | 일부 | 전체 |
| 보안 요구 수준 | 일반 | 중요 | 매우중요 |

### 5.2 비즈니스 복잡도 요소

| 요소 | 저 | 중 | 고 |
|------|:--:|:--:|:--:|
| 업무 프로세스 수 | <5 | 5~15 | >15 |
| 사용자 유형 | 1~2 | 3~5 | 6+ |
| 규제/컴플라이언스 | 없음 | 일부 | 필수 |
| 레거시 연동 | 없음 | 일부 | 많음 |

### 5.3 복잡도 점수 산정

```
총 복잡도 점수 = 기술 복잡도(40%) + 비즈니스 복잡도(40%) + 조직 복잡도(20%)

[점수 범위]
- 1.0~1.2: 낮은 복잡도 (보정계수 0.9)
- 1.2~1.5: 보통 복잡도 (보정계수 1.0)
- 1.5~2.0: 높은 복잡도 (보정계수 1.1~1.2)
- 2.0+: 매우 높은 복잡도 (보정계수 1.3+)
```

## 6. 프로젝트 유형별 템플릿

### 6.1 표준 프로젝트 템플릿

| 유형 | 규모(FP) | 기간(월) | 인력(명) |
|------|:-------:|:-------:|:-------:|
| 소형 웹앱 | 100~300 | 2~4 | 3~5 |
| 중형 웹앱 | 300~800 | 4~8 | 5~10 |
| 대형 시스템 | 800~2000 | 8~18 | 10~30 |
| 엔터프라이즈 | 2000+ | 18+ | 30+ |

### 6.2 프로젝트 유형별 특성

```
[B2B SI 프로젝트]
├── 특성: 대규모, 복잡한 요구사항
├── 기간: 6개월~2년
├── 팀 규모: 10~50명
└── 산정 방식: FP 또는 투입공수

[스타트업 MVP]
├── 특성: 소규모, 빠른 출시
├── 기간: 1~3개월
├── 팀 규모: 2~5명
└── 산정 방식: 기능 단위 패키지

[에이전시 프로젝트]
├── 특성: 중소규모, 표준화된 패턴
├── 기간: 1~6개월
├── 팀 규모: 3~8명
└── 산정 방식: 페이지/기능 단위
```
