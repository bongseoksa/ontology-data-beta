# 커뮤니티 핵심 페르소나- 기술 자랑이 아니라 ‘의사결정 구조를 어떻게 설계했는가’를 보여줘야 한다.

## 프로젝트
- 타겟: 중소기업 대표 + 일반인
- 목표: “우리는 AI를 깊게 이해하고, 실제 문제를 구조적으로 푼다”를 보여주는 것
- 수단: 가격산정 AI 서포터
- 차별점: 파운더리 온톨로지 기반
- 분석 대상: 소프트웨어 사업계획


## 허들 시발점
1. 파운더리 온톨로지에 사용될 데이터를 어디까지 정의할것인가 
- 산업군을 제한하여 관계성 딥하게 만들기. 이후 산업군 확장 
- 산업군을 제한하지 않고 범용적인 사용성을 제공. 이후 딥하게 관계성 보완

2. 소프트웨어 산업 한정으로 진행시 온톨로지 설계
- 기준이 되는 계층 기준

3. 온톨로지와 RAG 연동 계획
- llm 페르소나: 기술전문가, 사업전문가, 경제전문가

3. 제조업, 서비스업(식당 제외) 등 산업군 확장 방법 계획
- 산업 확장은 온톨로지 추가가 아닌, core 온톨로지는 유지한 채 core기반으로 확장해야하는건가 

## 생각나는 대로의 정보
1. 온톨로지 설계 관점에서의 소프트웨어의 정의
- 소프트웨어 구분을 sass, on-premise, hybrid로 하면 안돼는 이유?
- 그렇다면 소프트웨어를 어떻게 정의해야 하는가?
2. 소프트웨어 산업을 하는 중소기업에서 견적산정 유형?
- 나의 생각: b2b si 입찰, 정부과제 수주, 설루션 개발 외주, 에이전시 등
- 견적 산정 유형 기준이 온톨로지 설계 관점에 적합한가?
3. 온톨로지 구조화를 위한 정형화 방법은?

## 추천 구조
```
Company
  ├── hasProject → Project
  ├── adoptsPricingStrategy → PricingStrategy
  └── hasFinancialConstraint → CashFlow

Project
  ├── requiresResource → Resource
  ├── definesScope → Scope
  └── producesCost → CostStructure

Pricing
  ├── referencesProjectCost
  ├── appliesMargin
  └── adjustsByMarketCondition
```
