# Phase 2: 도메인 범위 정의

---

## 1. 산업 범위 (Industry Scope)

### 1.1 1단계: 소프트웨어 산업 한정

| 구분 | 포함 여부 | 비고 |
|------|:--------:|------|
| 소프트웨어 개발 | O | 웹, 모바일, 데스크톱 애플리케이션 |
| 시스템 통합 (SI) | O | 기업용 시스템 구축 |
| IT 컨설팅 | O | 기술 자문 포함 프로젝트 |
| 임베디드 소프트웨어 | △ | 2단계 확장 고려 |
| 하드웨어 제조 | X | 제외 |
| 건설/토목 | X | 제외 |
| 제조업 | X | 제외 (별도 온톨로지 필요) |

### 1.2 산업 확장 전략
```
[1단계] 소프트웨어 산업 Deep Dive
    ↓
[2단계] IT 서비스 확장 (클라우드, 보안 등)
    ↓
[3단계] 타 산업 확장 (제조, 서비스 등)
```

**핵심 원칙**: Core Ontology는 유지하고 도메인 확장 구조로 진행

## 2. 계약 유형 범위 (Contract Type Scope)

### 2.1 1단계 포함 범위

| 계약 유형 | 설명 | 포함 |
|----------|------|:----:|
| **Project-Based** | 프로젝트 단위 계약 (SI, 외주 개발) | O |
| **Bid-Based** | 입찰 기반 계약 (민간 입찰) | O |
| **Time & Material** | 투입 공수 기반 계약 | O |
| **Fixed Price** | 고정 가격 계약 | O |

### 2.2 2단계 확장 예정

| 계약 유형 | 설명 | 확장 시기 |
|----------|------|----------|
| **Government 과제** | 정부/공공기관 수주 | 2단계 |
| **R&D 과제** | 연구개발 과제 | 2단계 |
| **유지보수 계약** | 연간 유지보수 | 3단계 |

### 2.3 계약 유형별 특성

```
Project-Based (프로젝트 기반)
├── 범위: 명확한 요구사항
├── 기간: 정해진 일정
├── 가격: 범위 기반 산정
└── 리스크: 요구사항 변경

Bid-Based (입찰 기반)
├── 범위: RFP 기반
├── 기간: 발주처 지정
├── 가격: 경쟁 가격 책정
└── 리스크: 저가 수주
```

## 3. 제외 범위 (Exclusions)

### 3.1 명시적 제외 항목

| 항목 | 제외 사유 |
|------|----------|
| **순수 SaaS 구독 전략** | 프로젝트 기반이 아닌 반복 수익 모델 |
| **제조업 프로젝트** | 별도의 원가 구조 및 온톨로지 필요 |
| **건설/토목 프로젝트** | 산업 특성상 완전히 다른 비용 구조 |
| **하드웨어 제품 판매** | 소프트웨어 개발이 아닌 제품 판매 |
| **식당/요식업** | 서비스업이나 산업 특성 상이 |

### 3.2 경계 사례 (Edge Cases)

| 사례 | 판단 | 근거 |
|------|------|------|
| SW + HW 통합 프로젝트 | 부분 포함 | SW 개발 부분만 산정 |
| SaaS 초기 개발 | 포함 | 프로젝트 기반 개발 단계 |
| 마이그레이션 프로젝트 | 포함 | 기존 시스템 전환 |
| 운영 대행 | 제외 | 지속적 서비스 모델 |

## 4. 소프트웨어 분류 기준

### 4.1 온톨로지 관점의 소프트웨어 정의

**배포 방식이 아닌 "개발 특성"으로 분류**

| 분류 방식 | 적합성 | 이유 |
|----------|:------:|------|
| SaaS/On-premise/Hybrid | X | 배포 후 운영 방식 (가격 산정과 무관) |
| 기능/도메인 기반 | O | 개발 복잡도와 직접 연관 |
| 기술 스택 기반 | O | 인력 단가 및 난이도 영향 |

### 4.2 소프트웨어 유형 분류

```
[애플리케이션 유형]
├── 웹 애플리케이션
│   ├── 프론트엔드
│   ├── 백엔드
│   └── 풀스택
├── 모바일 애플리케이션
│   ├── iOS 네이티브
│   ├── Android 네이티브
│   └── 크로스 플랫폼
├── 데스크톱 애플리케이션
├── API/플랫폼
└── 데이터/AI 시스템

[시스템 유형]
├── ERP/CRM/SCM
├── 전자상거래 플랫폼
├── 콘텐츠 관리 시스템
├── 데이터 분석 플랫폼
└── IoT/임베디드 시스템
```

## 5. 견적 산정 유형

### 5.1 소프트웨어 산업 내 견적 유형

| 유형 | 설명 | 특성 |
|------|------|------|
| **B2B SI 입찰** | 기업 대상 시스템 구축 | 대규모, 장기, 복합 기술 |
| **정부과제 수주** | 공공기관 프로젝트 | 규정 기반, 공개 입찰 |
| **솔루션 개발 외주** | 특정 솔루션 개발 의뢰 | 중규모, 명확한 요구사항 |
| **에이전시 프로젝트** | 웹/앱 개발 대행 | 소규모, 단기, 반복적 |
| **스타트업 MVP** | 최소 기능 제품 개발 | 소규모, 빠른 출시 중요 |

### 5.2 견적 유형별 가격 산정 특성

```
B2B SI 입찰
├── 산정 방식: 기능점수(FP) + 투입공수
├── 참조 기준: 공공 SW사업 대가기준
├── 마진 구조: 원가 + 이윤율
└── 리스크: 요구사항 변경, 일정 지연

에이전시 프로젝트
├── 산정 방식: 페이지/기능 단위
├── 참조 기준: 시장 시세
├── 마진 구조: 패키지 가격
└── 리스크: 범위 확장 (Scope Creep)
```

## 7. 주요 의사결정 사항

### 7.1 결정된 사항

| 항목 | 결정 | 근거 |
|------|------|------|
| 초기 산업 범위 | 소프트웨어 한정 | 도메인 전문성 확보 우선 |
| 계약 유형 | Project/Bid 기반 | 가격 산정 로직의 일관성 |
| SW 분류 기준 | 개발 특성 기반 | 비용 영향 요소와 직결 |

### 7.2 향후 결정 필요 사항

| 항목 | 검토 시점 | 고려 사항 |
|------|----------|----------|
| 정부과제 포함 여부 | Phase 3 이후 | 별도 규정 및 단가 기준 |
| 유지보수 계약 포함 | Phase 6 이후 | 다른 비용 구조 |
| 타 산업 확장 | 1단계 완료 후 | Core Ontology 검증 필요 |
